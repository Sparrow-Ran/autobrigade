# CLAUDE.md — АВТОБРИГАДА (рабочее название)

## Что мы делаем

Кооперативная комедийная игра на 4 игроков (friendslop, в духе Lethal Company / Peak / R.E.P.O.), где команда управляет ОДНИМ фургоном, и у каждого игрока ровно ОДИН орган управления:

- Игрок 1 — рычаг ГАЗА
- Игрок 2 — рычаг ТОРМОЗА
- Игрок 3 — рычаг руля ВЛЕВО
- Игрок 4 — рычаг руля ВПРАВО

Игроки — физические персонажи, которые ходят по миру и по салону фургона, садятся на посты, встают, помогают друг другу. Веселье игры рождается из хаоса координации голосом: ехать можно только договариваясь.

Структура игры (дальняя цель): цикл квоты. Диспетчер выдаёт заказы на доставку → команда едет и везёт физически симулированный груз → оплата зависит от состояния груза и фургона → заработок тратится на ремонт/апгрейды ИЛИ копится на квоту, которую нужно сдать раз в 3 игровых дня. Провал квоты = конец забега.

## Ключевые дизайн-принципы (сверяйся с ними при любом решении)

1. **Провал должен быть смешным, а не фрустрирующим.** Физика валкая, но управляемая. Ошибки видны всем.
2. **Рычаги — театр.** Все органы управления — видимые 3D-рычаги в салоне, которые физически двигаются. Все игроки видят, что делает каждый.
3. **Аналоговость = скилл.** Рычаги имеют непрерывный ход (float 0..1), не кнопки. У рычагов ограниченная скорость хода — мгновенно перекинуть нельзя.
4. **Машина не знает про игроков.** Цепочка: игрок → персонаж → сиденье (слот) → рычаг → машина. Машина читает только состояния слотов.
5. **Никакой полировки раньше времени.** Персонажи — капсулы. Сначала проверяем, что ядро весело.

## Технологии и версии

- **Godot 4.7** (бинарник: `E:\Godot_v4.7-stable_win64.exe`), рендер Forward+, физика Jolt
- **Только GDScript**, никакого C#
- Мультиплеер: РЕАЛИЗОВАН — `ENetMultiplayerPeer` (порт 7777), хост-авторитарная модель: физика фургона/рычагов считается ТОЛЬКО у хоста, клиенты шлют input по RPC, состояние синкается `MultiplayerSynchronizer`'ами. План: миграция на Steam (GodotSteam, `SteamMultiplayerPeer` — drop-in замена; всё сетевое проходит через автолоад `Net`, менять придётся только его)
- Физика машины: `VehicleBody3D` + `VehicleWheel3D`
- Персонаж: `CharacterBody3D`, камера от ПЕРВОГО лица (от 3-го отказались на этапе 1)

## Архитектура

### Основные сущности

- `Net` (автолоад, `scripts/network_manager.gd`) — вся сессия: host/join/leave, спавн игроков (имя ноды = peer id, ник и цвет — spawn-синк), зачистка при дисконнекте. Dev-шорткат: `godot ++ --host` / `godot ++ --join <ip>` минует меню
- `Player` (CharacterBody3D) — капсула, ходьба, прыжок, RayCast-взаимодействие (E, группа `interactable`). Авторитет ввода — свой пир; свой меш скрыт через render-слой 2 + cull mask (другие видят). Умеет ехать стоя в кузове: переноска дельтой трансформа фургона (позиция полностью, поворот только yaw), в кузове позиция синкается в ЛОКАЛЬНЫХ координатах фургона
- `Seat` — слот в фургоне. `occupant_peer: int` (0 = свободен) — серверное состояние, синкается всем; посадка/вставание через RPC с валидацией хостом. Каждый пир сам приклеивает ЛЮБОГО сидящего к креслу локально (не по синку трансформа — иначе отставание на пинг). Камера рабочего вида с обзором «головой» (лимиты yaw/pitch)
- `Lever` — рычаг. `value` (0..1) симулируется ТОЛЬКО хостом (slew rate, spring), синкается вниз; клиенты шлют grab/target по RPC. Право дёргать: хозяин поста ИЛИ кто угодно, пока пост пуст («дотянуться» — и для соло-теста, и для хаоса). Типы: **hold** (газ, рули), **spring** (тормоз). Направление хода: FORWARD (газ/тормоз), LEFT/RIGHT (рули валятся вбок)
- `ToggleSwitch` — физический тумблер (сейчас один: ЗАДНИЙ ХОД у поста тормоза). Состояние синкается; переключение: R сидя или E стоя, право — как у рычагов
- `Van` (VehicleBody3D) — каждый physics tick читает ТОЛЬКО состояния слотов:
  - engine_force = газ * MAX_FORCE, знак от тумблера ЗХ (реверс слабее, `reverse_force_scale`); ВНИМАНИЕ: вперёд = ОТРИЦАТЕЛЬНАЯ сила (нос фургона в -Z)
  - brake = тормоз * MAX_BRAKE
  - steering = (лево − право) * max_steer * ослабление от скорости (`steer_falloff_kmh`)
  - На клиентах кузов заморожен (freeze), трансформ и ротации колёс приходят синком. Звуки (мотор с тоном от газа+скорости, клаксон) — процедурные placeholder'ы, считаются локально у каждого пира
- HUD (`scenes/ui/hud.tscn`, у локального игрока) — прицел, подсказка interact (`get_prompt()` у интерактивных объектов), F3 дебаг-панель, Esc меню паузы (продолжить / перевернуть фургон / выйти)

### Слои

- Физические: 1 — мир (земля, кузов, конусы), 2 — игроки, 3 (значение 4) — сиденья и тумблеры (чтобы кузов-ригид не толкался со своей же статикой; маска игрока и interact-луча = 5)
- Render-слой 2 — «своё тело»: каждый клиент переносит туда свой меш и вырезает его из всех камер

### Структура проекта

```

```

### Соглашения

- Нейминг: snake_case для файлов/переменных/функций, PascalCase для классов и нод
- Каждой значимой сцене — свой `class_name`
- Связь между системами — через сигналы, не через прямые обращения вглубь чужих деревьев нод
- Все «магические числа» физики и рычагов — в `@export`-переменных, их будем крутить в инспекторе при тюнинге веселья
- Input actions в Input Map, не хардкод клавиш: `move_*`, `jump`, `interact`, `lever_axis`, `reverse_toggle` (R), `horn` (H), `debug_panel` (F3)
- Комментарии и названия в коде — английские; тексты для игрока — пока русские
- Управление сидя: ЛКМ-драг — свой рычаг, свободная мышь — обзор головой; WASD/Space — соло-режим «дотянуться» до всех рычагов (двигает их target, slew rate сохраняется)
