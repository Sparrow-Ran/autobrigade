# CLAUDE.md — АВТОБРИГАДА (рабочее название)

## Что мы делаем

Кооперативная комедийная игра на 4 игроков (friendslop, в духе Lethal Company / Peak / R.E.P.O.), где команда управляет ОДНИМ фургоном, и у каждого игрока ровно ОДИН орган управления:

- Игрок 1 — рычаг ГАЗА
- Игрок 2 — рычаг ТОРМОЗА
- Игрок 3 — рычаг руля ВЛЕВО
- Игрок 4 — рычаг руля ВПРАВО

Игроки — физические персонажи, которые ходят по миру и по салону фургона, садятся на посты, встают, помогают друг другу. Веселье игры рождается из хаоса координации голосом: ехать можно только договариваясь.

Структура игры (дальняя цель): цикл квоты. Диспетчер выдаёт заказы на доставку → команда едет и везёт физически симулированный груз → оплата зависит от состояния груза и фургона → заработок тратится на ремонт/апгрейды ИЛИ копится на квоту, которую нужно сдать раз в 3 игровых дня. Провал квоты = конец забега.

## Ключевые дизайн-принципы (сверяйся с ними при любом решении)

1. **Провал должен быть смешным, а не фрустрирующим.** Физика валкая, но управляемая. Ошибки видны всем.
2. **Рычаги — театр.** Все органы управления — видимые 3D-рычаги в салоне, которые физически двигаются. Все игроки видят, что делает каждый.
3. **Аналоговость = скилл.** Рычаги имеют непрерывный ход (float 0..1), не кнопки. У рычагов ограниченная скорость хода — мгновенно перекинуть нельзя.
4. **Машина не знает про игроков.** Цепочка: игрок → персонаж → сиденье (слот) → рычаг → машина. Машина читает только состояния слотов.
5. **Никакой полировки раньше времени.** Персонажи — капсулы. Сначала проверяем, что ядро весело.

## Технологии и версии

- **Godot 4.x** (актуальная стабильная), рендер Forward+
- **Только GDScript**, никакого C#
- Мультиплеер (позже): `ENetMultiplayerPeer`, хост-авторитарная модель — физика фургона считается ТОЛЬКО у хоста, клиенты шлют input своих рычагов (float), получают трансформы
- Физика машины: `VehicleBody3D` + `VehicleWheel3D`
- Персонаж: `CharacterBody3D`

## Архитектура

### Основные сущности

- `Player` (CharacterBody3D) — капсула, ходьба, прыжок, камера от 3-го лица, RayCast-взаимодействие (клавиша E, объекты в группе `interactable`)
- `Seat` — слот в фургоне. Знает: занят/свободен, кем, какой `Lever` к нему привязан. Сажает/высаживает персонажа (фиксация + переключение камеры на рабочий вид)
- `Lever` — рычаг. Состояние: `value: float` (позиция), `target: float`, `slew_rate: float` (макс. скорость хода). Управляется мышью (drag) или осью. Типы поведения:
  - **hold** (газ): остаётся там, где оставили
  - **spring** (тормоз): подпружинен, отпустил — возвращается в 0
- `Van` (VehicleBody3D) — читает 4 значения рычагов из слотов каждый physics tick:
  - engine_force = газ (0..1) * MAX_FORCE
  - brake = тормоз (0..1) * MAX_BRAKE
  - steering = (лево - право), т.е. два руля ВЫЧИТАЮТСЯ — чтобы ехать прямо после дуги, двум игрокам надо балансировать друг против друга

### Структура проекта

```
res://
  scenes/
    player/      # персонаж, камера
    van/         # фургон, сиденья, рычаги
    levels/      # тестовый полигон и будущие карты
    ui/
  scripts/       # общие скрипты, автолоады
  assets/        # модели, материалы (пока плейсхолдеры)
```

### Соглашения

- Нейминг: snake_case для файлов/переменных/функций, PascalCase для классов и нод
- Каждой значимой сцене — свой `class_name`
- Связь между системами — через сигналы, не через прямые обращения вглубь чужих деревьев нод
- Все «магические числа» физики и рычагов — в `@export`-переменных, их будем крутить в инспекторе при тюнинге веселья
- Input actions в Input Map, не хардкод клавиш: `move_*`, `jump`, `interact`, `lever_axis`
- Комментарии и названия в коде — английские; тексты для игрока — пока русские
